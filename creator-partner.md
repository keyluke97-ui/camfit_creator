# 캠핏 파트너 협찬 — 크리에이터 신청 페이지 컨텍스트

> **이 문서의 목적**: 새 프로젝트 폴더에서 크리에이터 파트너 협찬 신청 기능을 개발할 때, AI 에이전트가 비즈니스 맥락을 파악하기 위한 참조 문서.
> **작성일**: 2026-04-02
> **작성 기준**: SDD v2 + 핸드오버 v5.3 + 캠지기 퍼널 핸드오버

> **v3 업데이트 (2026-04-22)**: 파트너 협찬 스키마를 등급 체계로 재편. 단일 `할인 금액`, 쿠폰 적용 요일 3옵션, 등급별 9필드 활성화. 자세한 내용은 `specs/partner-sponsorship-v3.md`.

## 1. 캠핏 파트너 협찬이란?

캠핏의 세 번째 크리에이터 협찬 라인. 캠지기(캠핑장 운영자)가 할인 쿠폰을 발행하고, 크리에이터가 캠핑장을 방문 후 콘텐츠를 제작하면서 팔로워에게 쿠폰을 배포하는 구조.

### 기존 협찬과의 차이

| 구분 | 프리미엄 협찬 (기존) | 파트너 협찬 (신규) |
|------|---------------------|-------------------|
| 제공 방식 | 출장비 + 숙박권 직접 지급 | 할인 쿠폰 발행 (팔로워 배포) |
| 수혜 대상 | 크리에이터 1인 | 크리에이터 + 팔로워 |
| 캠지기 부담 | 고비용 | 쿠폰 사용분만 할인 적용 |
| 협찬 금액 | 등급별 협찬 제안 금액 존재 | 협찬 금액 없음 (쿠폰 기반) |
| 참여 채널 | 블로그/인스타/유튜브 모두 | **인스타·유튜브만 (블로그 제외)** |

**블로그 제외 사유**: 쿠폰을 DM·스토리로 개별 전달해야 하는데, 블로그에는 그 메커니즘이 없음.

## 4. 비즈니스 플로우 — 크리에이터 신청 흐름

### 4.1 전체 흐름

```
크리에이터 로그인
  ↓
유저 테이블에서 채널 정보 확인
  ↓
채널이 인스타그램 또는 유튜브?
  ├─ YES → 상단 탭: [프리미엄 협찬] [캠핏 파트너] 노출
  └─ NO (블로그만) → [프리미엄 협찬]만 노출 (파트너 탭 숨김)
  ↓
파트너 탭 클릭 → /dashboard?tab=partner
  ↓
파트너 캠페인 목록 로드 (모집중 우선 → 최신순)
  ↓
캠페인 카드 클릭 → 신청 모달 열림
  ↓
정책 확인 플로우 (4단계):
  1. 방문 가능 기간: "YYYY.MM.DD ~ YYYY.MM.DD" 표시 → "이 기간 내에 방문 가능하신가요?" 체크
  2. 쿠폰 유효기간: "YYYY.MM.DD ~ YYYY.MM.DD" 표시 → "팔로워 쿠폰이 이 기간에만 사용 가능합니다" 체크
  3. 취소/변경 정책: "캠지기는 매칭 후 취소/변경 불가, 크리에이터는 방문일 전 취소/변경 가능" 체크
  4. 입실 희망일 선택 (방문 가능 기간 내 날짜만 선택 가능) + 입실 사이트 입력
  ↓
최종 리뷰 화면 (캠핑장명/패키지/할인금액/방문기간/입실일/쿠폰유효기간/취소정책 요약)
  ↓
신청 제출
  ↓
POST /api/partner/campaigns/apply
  - 사전 체크: 해당 등급의 `신청 가능 인원` (⭐️/✔️/🔥 중 본인 등급) + 중복 신청 확인
  - Airtable 레코드 생성
  - 사후 검증: 동시 신청 초과 시 롤백
  - 본인 등급 `신청 가능 인원` 0 → 해당 등급 마감 표시; 3개 등급 모두 0 → 캠페인 전체 자동 마감
  ↓
내 신청 목록에 반영
  ↓
운영팀이 Airtable에서 쿠폰 코드 수기 입력
  ↓
크리에이터가 포털에서 쿠폰 코드 확인 (있으면 코드 표시, 없으면 "준비 중")
```

### 4.2 탭 전환

- URL: `?tab=partner` 쿼리 파라미터로 관리 (`useSearchParams()`)
- 새로고침 시 유지, 링크 공유 가능
- 블로거가 `?tab=partner`로 직접 접근 시 → 프리미엄으로 폴백

### 4.3 등급(Tier) 시스템

프리미엄 협찬의 3등급을 파트너에도 동일 적용:

| 등급 | 이모지 접두사 | 등급별 차등 항목 |
|------|-------------|----------------|
| Icon (최상위) | ⭐️ | 쿠폰 수량, 모집 인원 |
| Partner (중간) | ✔️ | 쿠폰 수량, 모집 인원 |
| Rising (신규) | 🔥 | 쿠폰 수량, 모집 인원 |

> 파트너 협찬에는 협찬 금액이 없다. 등급별로 달라지는 건 쿠폰 수량과 모집 인원뿐.

### 4.4 캠페인 카드에 표시할 정보

- 캠핑장명
- 쿠폰 적용 요일: 아이콘 뱃지로 구분 (평일전용 / 평일+주말(금토) / 평일+주말+공휴일) — v3 단일 필드
- 할인 금액 (정액, 원 단위) — v3 단일 필드
- 팔로워용 쿠폰 수량 (등급별: ⭐️/✔️/🔥 쿠폰 수량)
- 모집 상태 (모집중/마감)
- 크리에이터 체험 쿠폰 포함 여부 (100% 할인)
- 방문 가능 기간
- 쿠폰 유효기간

### 4.5 취소/변경 정책

| 시점 | 캠지기 | 크리에이터 |
|------|--------|-----------|
| 신청~매칭 전 | 자유 취소/변경 | — |
| 매칭 후 | **취소/변경 불가** | 방문일 전까지 취소/변경 가능 |
| 불가항력 | 운영팀 판단으로 취소 가능 | — |

예약 취소/변경 요청은 카카오톡 채널(http://pf.kakao.com/_fBxaQG)로 유도.

## 6. 필요한 API 엔드포인트

| Method | Path | 설명 |
|--------|------|------|
| GET | `/api/partner/campaigns` | 파트너 캠페인 목록 (등급별 필터링, 모집중 우선 → 최신순) |
| POST | `/api/partner/campaigns/apply` | 파트너 캠페인 신청 (2단계 검증 + 자동 마감) |
| GET | `/api/partner/applications/my` | 내 파트너 신청 목록 (쿠폰 코드 포함) |
| PATCH | `/api/partner/applications/checkin` | 체크인 정보 수정 (입실일 + 입실 사이트) |
| PATCH | `/api/partner/applications/status` | 예약 상태 변경 |

모든 API에 JWT 검증 + 채널 기반 접근 제어 필수.

## 8. 핵심 비즈니스 규칙

### 8.1 블로거 제외 (이중 방어)
- **프론트**: 유저의 채널 종류에 `인스타` 또는 `유튜브`가 없으면 파트너 탭 자체 미렌더링
- **API**: 파트너 API route에서도 채널 기반 접근 제어 (보안 레이어)
- 블로그+인스타 복수 채널 → 파트너 탭 노출 (하나라도 인스타/유튜브면 OK)

### 8.2 동시 신청 방어 (Race Condition)
프리미엄 협찬과 동일한 2단계 검증:
1. **사전 체크**: 신청 가능 인원 확인 + 중복 신청 여부 확인
2. **Airtable 레코드 생성**
3. **사후 검증**: 동시 신청으로 초과 시 롤백

### 8.3 자동 마감 전환
- 특정 등급의 신청 가능 인원이 0 → 해당 등급 "마감" 표시
- **전 등급의 신청 가능 인원이 모두 0** → 캠페인 전체 `모집 상태`를 `마감`으로 변경

### 8.4 쿠폰 코드 표시
- 쿠폰 코드는 **캠페인 테이블**에 운영팀이 수기 입력 → 신청 테이블에서 Lookup으로 참조
- 포털에서는 읽기 전용 표시만 함
- 값이 있으면 → 코드 노출 + 복사 버튼
- 비어있으면 → "쿠폰 코드 준비 중" 표시, 복사 버튼 비활성화

### 8.5 공휴일 처리
- v3에서는 `쿠폰 적용 요일` 단일 필드로 통합됨 (3옵션: `평일전용` / `평일+주말(금토)` / `평일+주말+공휴일`)
- `평일+주말+공휴일` 선택 캠페인: 공휴일 **전일·당일** 입실 시 할인 적용
- 나머지 옵션 캠페인: 공휴일에 할인 미적용

### 8.6 할인 방식
- **정액(원) 확정**. 정률(%) 미사용.
- v3 단일 `할인 금액` 필드: 1~5만원 범위 내 캠지기 자유 설정 (평일/주말 구분 없이 단일 금액)
- 적용 요일은 `쿠폰 적용 요일` 필드(3옵션)로 별도 제어
- 평일 = 일~목 입실 / 주말 = 금·토 입실

## 12. 개발 제약사항

1. 기존 프리미엄 협찬 코드를 직접 수정하여 파트너 로직을 합치지 말 것 → 파트너 전용 컴포넌트·API를 별도로 만든다
2. Airtable 호출을 API route 파일에서 직접 하지 말 것 → `lib/airtable.ts`에 함수 추가 후 호출
3. `filterByFormula`에 사용자 입력 직접 넣지 말 것 → `escapeAirtableValue()` 필수
4. 디버그용 API route나 console.log 남기지 말 것
5. JWT 시크릿 하드코딩 금지
6. 쿠폰 발행·추적 시스템은 범위 밖 → 포털 UI + Airtable 읽기만
7. 블로거 제외 로직을 프론트엔드에서만 처리하지 말 것 → API에서도 검증 (이중 방어)
8. 함수 1개당 30줄 이내
9. Airtable 필드명은 한국어 유지 (영문 변경 시 매칭 실패)
10. 컴포넌트 파일명 PascalCase, 변수명 축약 금지
